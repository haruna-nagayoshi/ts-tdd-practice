# ts-tdd-practice

ECサイトの価格計算ロジック（送料・ポイント・割引） をTypeScriptで実装し、TDDと同値クラス分割・境界値分析を習得するプロジェクト。
さらに、知らないドメイン(残業代の計算)をAIで学習しながらTypeScriptで実装し、同様にTDDと同値クラス分割・境界値分析を習得する。

## 技術スタック

- TypeScript
- Vitest
- GitHub Actions

## カリキュラム（5セッション × 30分）

- Session 1: 環境構築 + 送料計算（同値クラス・境界値の体感）
- Session 2: ポイント計算（会員ランク × 誕生日月）
- Session 3: AI比較実験（丸投げ指示 vs 同値クラス表を渡した指示）→ **完了**
- Session 4: 残業代ドメインをAIで学ぶ（コードなし・表作成）
- Session 5: 残業代をTDDで実装

## 開発ワークフロー

- TDD: テスト先行 → 実装 → テストパス確認 → commit
- 1機能ずつ完成 → 動作確認 → 次へ のサイクルで進める
- commit・PRは必ずユーザーの許可を得てから実行する
- commitの前に必ず `npm run test` を実行してパスを確認する

## ユーザー情報

- 説明はTypeScriptの学習を兼ねて丁寧に行うことを好む
- PHPメインのエンジニア。TypeScript・Vitestは学習中

## Session 3 実験結果と学び

### 実験概要
割引計算（`calcDiscount`）を題材に、2種類のAI指示を比較した。

| | 丸投げ指示 | 同値クラス表あり指示 |
|--|-----------|-------------------|
| テスト数 | 6件 | 8件 |
| 代表値のテスト | なし（境界値のみ） | あり |
| 構造 | フラット | クラスごとに `describe` で整理 |
| 仕様への追跡性 | 低い | 高い |

### 学び
- 単純な仕様では丸投げでも境界値に気づく場合があるが、**代表値が抜ける・構造がない**という問題が残る
- **AIに先に同値クラスと境界値を分析させてから、その結果をもとにコード生成させると精度が上がる**
- テストが仕様に追跡できると、仕様変更時にどのテストを修正すべきか分かりやすくなる
